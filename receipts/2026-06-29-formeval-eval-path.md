# Receipt — form-eval wired as an EVAL PATH (`--feval`): recipes run THROUGH Form (2026-06-29)

**What happened:** `fkwu` gained a new, opt-in evaluation mode — `./fkwu.exe --feval recipe.fk` — that runs a
recipe THROUGH `form-eval` (a Form meta-evaluator), not through `fk_walk` (the C tree-walker) directly. The C
seed bootstraps `form-eval`; **`form-eval` runs the recipe**. This is the architectural completion of "get the
recipes off the C seed": the recipe is now interpreted by Form, which itself runs on `fk_walk`. The default
`--src`/table paths are byte-behavior-identical — this is a pure ADD.

## How `--feval` works

`fk_run_feval(path)` in `runtime/fkwu-uni.c`:

1. Reads the recipe source string from `path`.
2. Builds a single `--src`-shaped program in `fk_srctext`:
   `<embedded form-eval> + "\n(fe-eval \"<escaped-recipe>\")\n"`.
   - The embedded form-eval is `runtime/fkwu-formeval.h` — `grammars/form-eval.fk` plus the two helpers its
     `core.fk` prelude would supply (`char_at`, `ord`), GENERATED by `scripts/gen_formeval_bundle.py` so the
     embed stays faithful to the source recipe (regenerate after editing `form-eval.fk`; do not hand-edit).
   - The recipe is escaped into a Form string literal (`\` → `\\`, `"` → `\"`, newline → `\n`, CR dropped),
     so `fe-str` scans it as one literal regardless of the recipe's own line structure.
3. Runs the SAME `fk_prescan_defns` → `fk_parse_top` → `fk_walk` pipeline as `fk_run_src`. The root form is
   `(fe-eval "...")`; its value is the meta-eval's result — computed by `fe-eval`, in Form, on `fk_walk`.
4. Prints the result with `fk_pv` (value-kind: int / float / nothing). This sidesteps the print caveat:
   `fk_pv_root` keys string-vs-number off the *root op*, and `(fe-eval ...)`'s body chain ends in `head`
   (a non-string tag), so it already falls to the integer printer — but `fk_pv` makes the numeric scope
   explicit and correct. (A recipe that returns a *string* is outside this mode's numeric scope.)

Self-contained: the embedded form-eval means `--feval` needs no runtime file dependency — sovereignty-shaped,
the meta-evaluator ships inside the C-bootstrapped kernel.

## Measured slowdown (Deliverable 2) — honest

`--feval` is a two-level interpretation (Form meta-eval running on the C walker), so it is MUCH slower than
the direct C walk. Full-process wall times, Windows, TDM-gcc `-O2`:

| recipe | `--src` (direct `fk_walk`) | `--feval` (through form-eval) | slowdown |
|--------|----------------------------|-------------------------------|----------|
| `sum 5000` (linear) | ~0.27 s/run (~0.19 s compute) | ~2.59 s/run (~2.51 s compute) | **~13× (compute)** |
| `fib 22` (recursive) | ~0.038 s/run (compute sub-ms) | ~5.51 s/run | **~140×** |
| `fib 24` (recursive) | ~0.057 s/run (compute sub-ms) | ~14.9 s/run | **~260× raw wall** |

Process-startup floor (recipe `1`): `--src` ≈ 0.059 s, `--feval` ≈ 0.049 s — parsing the ~18 KB embedded
form-eval each run is cheap; the cost is purely the meta-interpretation. The slowdown GROWS with call depth:
every recipe call/op in form-eval is itself many `fk_walk` calls (string-compare op dispatch in `fe-disp`,
env assoc-list walks in `fe-look`, `(value end-pos env)` triple cons/head/tail per node). For linear work
it is ~13×; for deep recursion it is two-to-three orders of magnitude. **This slowdown is exactly what the
self-JIT (Deliverable 3) exists to close.**

## Self-JIT assessment (Deliverable 3) — what would make `--feval` the DEFAULT

There are TWO distinct JIT lanes; naming them precisely is the whole assessment.

### Lane A — the in-process self-JIT in the running `fkwu.exe`: **OPERATIONAL as a dispatch wire + HAL
carrier; SCAFFOLD as a general lowerer.**

- `fk_native_call` (`runtime/fkwu-uni.c`, Win64 `VirtualAlloc`+`VirtualProtect`, SysV `mmap`) is REAL and
  WITNESSED: it installs a lowered byte image as executable and jumps to it. The heat gate
  (`fk_heat[f]` / `fk_hot` from `argv[4]` / `fk_njit`) flips dispatch from the walker to native at threshold.
- **Witnessed live:** table mode `lit-42.flat` arg=7, cold (`hot=0`) → 42 (real fn0 body); hot+`j`
  (`hot=1`, `argv[5]='j'`) → **8** (= 7+1) with **njit=1**. The flip fires; native bytes ran instead of the
  walk. This proves the install+call+dispatch carrier end-to-end on real metal.
- **But the bytes are HAND-SUPPLIED.** The only native registered in the running kernel is `fk_demo_inc`
  (`mov rax,rcx; add rax,1; ret`), wired to fn0 only when `argv[5]=='j'`. A grep of `runtime/fkwu-uni.c` for
  any node-tree→bytes lowerer (`fk_nat_expr` / `fk_lower` / `fk_emit_native` / `fk_nat_tab` / `fk_jadd`…)
  returns **0**. The running kernel CANNOT take an arbitrary `--src` function's node tree and produce its
  native bytes. So general `--src`-function → asm crystallization is **NOT operational inside `fkwu.exe`** —
  the wire is live, the lowerer-behind-it is a single demo.

### Lane B — the Form emitter (`fkc-nat-expr` / `fkc-emit-jit2` in
`form/form-stdlib/hati-os-kernel-emit.fk`, decision in `observe/jit-decision.fk`, byte encoders in
`model/form-asm-x64.fk`): **a real recipe-tree → native lowerer, but ahead-of-time re-emit, not in-process.**

- `fkc-nat-expr` walks a recipe's node tree tag-by-tag (the pure-compute family, tags 1–7,12) and lowers it
  to a native body; `receipts/2026-06-29-jit-live-crystallize.md` measured `fib` crystallizing to native at
  a heat threshold with bit-identical output and **~13× speedup** (fib 30: 64.8 ms walk → 6.5 ms native).
- `model/form-asm-x64.fk` IS a Form recipe that emits real x64 instruction bytes (mov/add/sub/imul/ret) for
  the affine/arithmetic family — the right shape to feed `fk_native_call`. `observe/jit-decision.fk` is the
  pure-Form, four-way-proven crystallize/melt decision (hot≥5 AND pure → crystallize; heat<2 → melt;
  hysteresis so it never thrashes).
- **But Lane B works by re-emitting the ENTIRE walker as new C source** (with `fk_nat_tab[f]` dispatch),
  flattening via `bin-go`, and recompiling with `clang -O2` into a separate `fkjit` binary. It is an
  emit-and-recompile, NOT a JIT inside the running `fkwu.exe`. It also currently rides the `bin-go` +
  `clang` bootstrap (the audit's clang-compiled self-JIT C is the speed-reference rung; the sovereign target
  is the same recipe lowered Form→asm BYTES via `form-asm`, which the byte-identity gate proves four-way).

### What is precisely missing for form-eval to crystallize (make `--feval` the default)

The pieces exist; the GLUE inside the running kernel does not. To crystallize form-eval's hot functions
(`fe-disp`, `fe-eval`, `fe-look`, `fe-expr`, …) live, the running `fkwu.exe` needs:

1. **An in-process tree→bytes lowerer** — port `fkc-nat-expr` + `form-asm-x64.fk`'s byte encoders into (or
   callable from) the running kernel, so a hot `--src`/`--feval` function's node tree is lowered to bytes at
   runtime — not re-emitted as C and re-clanged. Today Lane A has the install door but no lowerer; Lane B has
   the lowerer but only ahead-of-time.
2. **Registration wiring** — when `fk_heat[f] >= fk_hot` and the decision (`jit-decision.fk`) says
   crystallize, lower fn `f`'s tree, install the bytes via `fk_native_call`, and set `fk_nat_code[f]`/
   `fk_nat_len[f]` so the next dispatch flips. Today only fn0 with hand-bytes is registrable.
3. **Coverage beyond pure-arithmetic** — form-eval's hot path is dominated by STRING ops (`str_eq` dispatch
   in `fe-disp`) and LIST/cons ops (env assoc-lists, `(value end-pos env)` triples), NOT the int-arithmetic
   family `fkc-nat-expr`/`form-asm-x64.fk` lower today. Crystallizing form-eval needs the lowerer extended to
   the string/list/cons tags (or form-eval's hot inner loops refactored toward the lowerable family). This is
   the substantive gap: the demonstrated ~13× crystallization is the int pure-recursion family; form-eval's
   bottleneck is string/list-shaped.

## Honest floor

`--feval` is now a REAL eval PATH: recipes run through Form (`form-eval`), proven end-to-end on real metal
(6, 720, fib10=55), value-identical to the direct walker, existing paths unregressed. It is **opt-in, not the
default** — the default stays `fk_walk` because `--feval` is ~13×–260× slower (two-level interpretation).
Making it the default awaits the self-JIT: the install+dispatch carrier is live and witnessed, a recipe-tree
→ native lowerer exists and crystallizes the int pure family ~13×, and the crystallize decision is four-way
proven — but the in-process glue (runtime lowerer + registration) and string/list-op coverage are the named,
specific gaps. The pieces are real; the wire from form-eval's hot cells to live native is the next witness,
not this one. Platform receipt rows (mac, android) are pending; this witness is Windows.
