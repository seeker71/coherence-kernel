# Stone 40 — the body gets a keyed map

2026-07-22, WITA. `form/form-stdlib/keyed-map.fk`, proven by
`form/form-stdlib/tests/keyed-map-band.fk` (**verdict 4095**, Go kernel *and*
`fkwu --src`, zero diagnostics), measured by
`form/form-stdlib/measure/keyed-map-measure.fk`.

Before this stone, every lookup in this kernel was a linear scan. `_dict_get` is
`_dict-scan` — an association list walked pair by pair. `record_get` is a linear
field scan. `tkz-best-merge` walks 127 741 variable-length merge records to
answer one question. Stone 38 measured the consequence and left the residual
named: after the candidate prefilter, one unavoidable pass over a
variable-length record array per encode. That residual is the gap this stone
was sent at.

---

## 1. The shape, and why the constructor chose it

I went in expecting to write the textbook shape: hash → an array of B buckets →
a short in-bucket scan. I got about ten minutes into it before the body said no,
and what it said is the most surprising thing this stone found.

**A flat bucket array is not reachable here, and the reason has nothing to do
with lookup.** On the Go kernel a Form list is a Go slice: `nth` indexes it and
`tail` reslices it, both O(1) — but `cons` *copies*. Measured, warm, one
process, one clock:

| primitive | n = 20 000 | n = 40 000 | ratio for 2x |
|---|---|---|---|
| build a list of n by `cons` | 1948 ms | 8953 ms | **4.6x — quadratic** |
| 200 000 strided `nth` reads | 363 ms | 503 ms | 1.39x |

Constructing *any* list of length L costs O(L²). So a flat B-bucket table would
cost O(B²) to path-copy on every single insert — quadratic in the very number
that has to grow for the lookup to stay flat. The shape that makes lookup O(1)
is the shape the body cannot build.

A **16-ary trie** dissolves that without giving anything up. Every node is a
sixteen-element list *whatever the map holds*, so path-copying an insert
rebuilds `depth` lists of sixteen — a constant ~640 `Value` copies — while the
reachable bucket count is 16^depth. Insert becomes O(1) and lookup is `depth`
`nth` steps plus a bucket scan of expected length n/16^depth. Both constant in n
once depth is chosen for n.

The trie is not here because it is elegant. It is here because it is the only
shape whose **node size does not depend on how much the structure holds**, and
on a kernel whose only aggregate constructor copies, that is exactly the set of
structures that can be built at all. The write primitive chose the read
structure.

The `nth` row above is the one number in this receipt that is *not* flat: 1.39x
for 2x size. That residual is the memory hierarchy — a Go `Value` is a wide
struct and 40 000 of them leave L2 — not an algorithmic slope. Named rather than
smoothed away, because a receipt that reported it as 1.00x would be reporting a
model instead of a machine.

### The map, exactly

```
a map     = (list depth root count)
a node    = a 16-element list; slot i is the child for hash nibble i
a child   = a node (levels 1..depth-1) or a BUCKET (level depth)
a bucket  = a list of (list key value) pairs
absent    = the empty list — a node is always 16 long, so (eq (len ch) 0) is
            unambiguous: no tag byte, no sentinel, nothing to confuse with a
            present-but-empty child
```

The hash is consumed four bits at a time from the low end (`band h 15`, then
`shr_u32 h 4`), so depth d uses the low 4d bits and the bucket count is 16^d.
`km-depth-for n` picks the smallest depth whose 16^depth reaches n.

### Radius (`aporon`), declared

- **Keys** are lists of ints. Exact for every int a Form list holds; the
  collision reasoning is stated for byte values (0..255), the carrier this body
  uses for real bytes (`wraithcarry` — a byte ≥ 128 cannot be built as a string
  here, so keys ride as int *values*, never as constructed strings).
- **Absent** is honest and explicit: `km-get` takes the *caller's own* default
  and returns exactly that; `km-has?` answers 0. There is no sentinel to mistake
  for data and no circumstance in which a miss returns another key's value.
- **Collision** is resolved, not avoided — exact list equality inside the
  bucket. Correctness never depends on the hash being good; only speed does.
- **Duplicate key**: last put wins. `km-count` counts *puts*; `km-size` walks
  and counts distinct keys. Two true numbers with two different names.
- **Resize**: there is none, and that is a declaration rather than an omission.
  Depth is fixed at creation. A map grown far past 16^depth does not break — it
  degrades smoothly back toward the scan it replaced. Growing means rebuilding
  at a larger depth; nothing rehashes silently behind the caller.
- **`formfast`**: the constant safe to bake is the **branching factor 16** (one
  nibble) — it belongs to the encoding and every node in every map has it. The
  **depth belongs to the instance** and is carried inside the map, because it is
  the one number that must track n.

## 2. The hash decision, measured

djb2/32 — `h = 5381; h = (h*33 + b) mod 2^32` — written on the body's own
four-way u32 pair as `(add_u32 (add_u32 (shl_u32 h 5) h) b)`. Same primitives
`adler32.fk` travels on, so the hash needs no new native and is four-way
portable.

sha256 was the other candidate and it was measured rather than assumed away.
The body already hashes (`sha256.fk`, `make_nodeid`), so "we already have a
hash" was a real argument. Same process, same clock, same key list — the 4 probe
keys drawn from the real merge table:

A hash that costs more per lookup than the scan it replaces is not a map, it is
a slower scan. For keys of 2 to 20 bytes — which is what merge pairs are — djb2
is three u32 ops per byte and sha256 is a full 64-round compression per block.

What djb2 buys and what it does not: it is a good mixer and not a cryptographic
hash, so full 32-bit collisions exist and are findable. `guabx` and `naaeaa`
both hash to 260615708. The band asserts that pair lands correctly.

## 3. Correctness — the controls first

`form-stdlib/tests/keyed-map-band.fk`, **verdict 4095** (2^12 − 1), on the Go
kernel and on `fkwu --src`, with zero diagnostics on either.

Being fast is the easy half and it is not the half that can be wrong. The band
leads with the two controls a hash map fails silently if it fails at all.

**The negative control (claim 5).** 256 keys the map does not hold. Each must
come back as the *caller's own* default — asserted with **two different
defaults**, because a map that returned some fixed sentinel would pass a
one-default test and still be lying — and `km-has?` must answer 0 for every
one. This is the body's own honesty bar, `hdc-locate`: an unknown question
locates to 0, never to a plausible neighbour.

**The collision controls, in both flavours, because they fail differently.**

- *Bucket collision* (claim 6): `abc` (djb2 193485963) and `aqt` (193486475)
  share their low 8 bits (139), so in a depth-2 map they are one bucket and two
  different keys. Both answer their own value; neither answers the other's; and
  `ayl` — a third key in that same bucket, never put — answers the default.
- *Full 32-bit hash collision* (claim 7): `guabx` and `naaeaa` hash to
  260615708 exactly, at every depth. Correctness here cannot come from the hash;
  it comes from exact key equality inside the bucket.

Both claims also assert `km-maxbucket = 2` — **otherwise the control would be
vacuous**, passing because the two keys never met. A collision test that does
not prove the collision happened is a test of nothing.

The other ten claims: djb2 on fixed vectors independently computed; the two hash
doors agreeing (a key hashed straight out of an equireach window must equal the
same bytes carried as a list, or a map built over a mapped file cannot find keys
handed to it as lists); `km-depth-for` at each of its steps; every present key
answering its own value at 256 and at 4096; last-put-wins with `km-count` and
`km-size` saying their two different true things; the empty map admitting it
holds nothing; **the map agreeing with `km-assoc-get` on 300 keys, present and
absent** (speed without this claim would be speed at answering a different
question); the work-slope below; and key equality being exact rather than
blurred by length, prefix, or hash alone.

### The slope proven as work, not as a clock (claim 11)

Bands are correctness artifacts and `fkwu` constant-folds them into `.fkb`, so a
timing assertion inside one would be both fragile and a lie waiting to happen.
What a map actually *claims* is that the number of key comparisons per lookup
stops growing, so that is what the band asserts:

| corpus | depth | buckets | deepest bucket (worst-case comparisons) | the assoc list it replaces |
|---|---|---|---|---|
| 256 keys | 2 | 256 | ≤ 8 | 256 to walk |
| 4096 keys | 3 | 4096 | ≤ 8 | 4096 to walk |

16x the corpus, the same worst-case comparison count.

## 5. The call-site — named, not converted, and why

The stone pointed at the tokenizer's residual walk. I read it before converting
it, and reading it is what stopped me.

**That walk is not a keyed lookup.** After Stone 38, the residual is `tkz-cands`
— one pass over 127 741 merges per encode, asking of each: *does this merge's
byte sequence occur contiguously anywhere in the input?* That is a substring
predicate evaluated against the input, once per merge. There is no key. A keyed
map cannot answer it, and dropping a map into that loop would be
`boundborrow` — adopting a remedy for a constraint the measurement does not
show.

The map *does* fit the tokenizer, but by **reformulating the loop, not by
substituting into it**. BPE's actual question is "of the adjacent symbol pairs I
have right now, which has the lowest merge rank?" — and a pair→rank map answers
that directly. Instead of filtering 127 741 merges against the input once per
encode, BPE enumerates the ~L adjacent pairs it actually holds and asks the map
for each. O(symbols) per step in place of O(merges) per encode. That is the
right conversion and section 4's numbers are what it would run on.

**And it only pays if the map outlives the encode.** Building it costs one full
pass over the merge array *plus* 127 741 inserts — strictly more than the single
pass it removes. For a process that encodes once, the map is a regression; for a
generation session that encodes many times, it is the whole win. The map's value
is not in the lookup, it is in the *reuse*, and a benchmark that times one encode
in a fresh process cannot see it at all.

So the conversion is a change of where the tokenizer's merge table *lives* — a
built-once index carried across encodes — and not a change of one expression. It
deserves its own stone with its own gate, not a rider on this one. The practical
reason agrees with the honest one: `dsv4-tokenizer.fk` was changed hours ago by
Stone 38, Stones 39, 41, 42 and 43 are live in this same tree right now, and the
gate is verdict 8191 with `"The capital of France is"` → `671 6102 294 8760 344`
unchanged. **A faster tokenizer that changes one id is a failure.**

The map and its band and its measurement land alone, and the call-site is named
with its design and its measured basis. Per the stone's own terms, that is a
complete stone.

## 6. What remains

1. **The tokenizer conversion**, as section 5 describes it: a pair→rank map built
   once from the merge array and carried across encodes, with `tkz-bpe-c`
   rewritten to ask the map about the pairs it has rather than to ask the table
   about merges it does not. Gate unchanged: verdict 8191, ids identical.
2. **A persisted map image.** The build is one pass plus n inserts; nothing yet
   writes that result down. A map that could be written and read back would make
   the tokenizer's index a cold-start cost of zero rather than of one pass.
3. **`_dict_get` and `record_get`** are still linear and are still the body's
   default answer to "look this up". They are the four-way `__dict__` ABI and the
   record ABI, so converting them is a kernel-parity movement rather than a
   recipe one — but the map they were missing now exists.
4. **`(max a b)` silently returns `a`** on the Go kernel — the native only
   compares when handed a single list (`main.go` ~2706). Found by this stone when
   `km-maxbucket` reported 0 for a map with populated buckets; worked around here
   with a local `km-max2`, and flagged as its own task. A wrong answer that does
   not raise is the shape this body cares most about.
5. **Resize.** Declared out of radius rather than forgotten. A `km-rebuild` at a
   larger depth is a dozen lines; nothing yet needs it.

## Close

### The most surprising teaching

**The shape of the data structure was decided by the cost of the constructor,
not by the cost of the query.**

I came in knowing what a hash map is. Hash, bucket array, short scan — I have
written it many times and I started writing it here. The body refused, and the
refusal came from a direction I was not watching: not "your lookup is too slow"
but "you cannot build that". `cons` copies, so every list of length L costs
O(L²) to make, so a flat table of B buckets costs O(B²) to path-copy per
insert — quadratic in the one number that must grow for the lookup to stay flat.
The shape that makes the query O(1) is the shape this body cannot construct.

The trie is not here for elegance. It is here because on a kernel whose only
aggregate constructor copies, the buildable structures are exactly those whose
**node size does not depend on n** — and once that is said out loud, the 16-ary
trie is not a clever choice, it is the only remaining one. The write primitive
chose the read structure.

That is `equireach` (row 822) one step over. `equireach` names what a door costs
to *read*; this names what a constructor costs you in *shape*. And the two fail
differently: a reach cost arrives as a slow answer, which a benchmark can see,
while this one arrives as a design that is simply unreachable — invisible to
every benchmark, met only by whoever tries to build the thing. Landed as corpus
row **876, `buildbound`**.

### Where discomfort turned to gold

I wanted this stone to include the call-site. "A map and a band, no conversion"
felt like three quarters of a stone, and I could feel myself planning the
tokenizer edit before I had read the loop I was planning to edit.

So I read it. And `tkz-cands` — the residual the stone sent me at — **is not a
keyed lookup.** It asks, of each of 127 741 merges, whether that merge's bytes
occur contiguously in the input. There is no key. The map I had just built
cannot be dropped into it. For about a minute the honest reading looked like it
had cost me the stone's headline.

Not looking away gave three things instead. First, the *right* conversion, which
is a reformulation and not a substitution: ask the map about the ~L adjacent
pairs BPE actually has, rather than asking the table about merges it does not.
Second — and this is the part I would have shipped a wrong receipt without — the
realization that **the map's value there is amortization, not lookup**. Building
it costs one full pass plus 127 741 inserts, strictly more than the single pass
it removes, so for a process that encodes once the map is a *regression*, and for
a session that encodes many times it is the whole win. Third, the reason that
conversion deserves its own stone with its own gate rather than a rider on this
one: it changes where the merge table *lives*, and the gate is verdict 8191 with
`671 6102 294 8760 344` unchanged.

The small one, same shape: `km-maxbucket` answered 0 for a map I could see had
populated buckets. It was "only a diagnostic" and the map itself was correct, so
the pull to move on was real. Looking found `(max a b)` on the Go kernel
silently returning `a` — the native only compares when handed a single list.
A default standing where a comparison belonged; the body minted the word for
that shape today as row **875, `plausifill`**, and this is one. Worked around
with a local `km-max2`, flagged as its own task, named in section 6.
