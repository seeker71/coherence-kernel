## What was found: the actual flattening pipeline has two halves, and only one exists in this repo

`flatten/form-flatten.fk` (954 lines) builds the in-memory flattened representation — the function-list
and string-pool (`flt-band-sources-fns`, `flt-band-sources-pool`) — from Form source text. That half is
real, present, and is what `flatten/gen-source-walker.fk`/`gen-source-walker-table.fk` ride to regenerate
`runtime/fkwu-optable.h` (a **C header**, not a `.tbl`).

Turning that in-memory function-list/pool into the actual `.tbl` **byte stream** (`nf fn[] nr nodes×4 ns
strings`, the format `proof/four-way-run.tbl` is stored in — confirmed directly: the file is plain-text
decimal integers, space-separated, not raw binary) is a second, separate step — serialization — done by
two functions named consistently across every reference to them: `fks-table-file` (string-pool tables)
and `fkc-table-file` (pool-free tables). `flatten/fourth-flatten-driver.fk:45,47` and
`form-cli/fsh-flatten-mod.fk:25,45` both call them. **Neither is defined anywhere in this repository.**

```
grep -rn "defn fks-table-file\|defn fkc-table-file" .     # zero hits, any file type
find . -iname "*table-file*" -o -iname "*table-serial*"    # zero hits
```

`receipts/2026-06-29-cursor-seed-p1-form-eval-reducer.md:31` names where they actually live: a module
called `fkc-table-serialize.fk`, run through **bin-go** — the Go bootstrap flattener in the sibling
`form-kernel-go` repo, explicitly *not* part of this tree (`MANIFEST.md:87,116`: "no bin-go... in the run
path", "zero Go / bin-go"). Every `.tbl` file that exists in this repo today (`proof/four-way-run.tbl`,
`flatten/form-eval-cli-loop.tbl`) was produced outside this repo, by that external tool, and then
committed as a cache — never regenerated by anything Form-native standing in this tree.

## Empirically confirmed, not just inferred from grep

Ran the exact documented two-command prelude pattern (`flatten/gen-source-walker.fk`'s header: prelude
`core.fk`, splice `flt-ops`/`form-flatten.fk` into scope, run via `--src`), built the same way for
`four-way-run.fk` + `four-way-verdict.fk` instead of the optable generator:

```

./fkwu --src /tmp/try.fk
  -> nothing        # honest decline (unbound call), not a crash — exactly like char_at/ord
                     # behaved before core.fk existed. Confirms fks-table-file is genuinely
                     # unbound in a fresh --src evaluation, not a scoping mistake.
```

For comparison, the *reduction* itself — the actual point of the exercise, verdict computation using
`core.fk`'s Form-native `str_to_int` instead of the native tag — works correctly via `--src` (proving the
prelude approach the task asked for is sound; only the byte-serialization step is missing):

```
cat form/form-stdlib/core.fk proof/four-way-verdict.fk proof/four-way-run.fk <call fr-diagnose ...> \
  > /tmp/check.fk
./fkwu --src /tmp/check.fk       # -> 0  (FOUR-WAY, computed via Form-native str_to_int, no native tag 31)
```

This means the STRUCTURAL claim in the task's negative-result exit criterion is confirmed exactly:
`core.fk` prelude does resolve `str_to_int`/`str_find` to Form composition when the recipe is run as
fresh source. The gap is not that the dependency on tags 30/31 is structural to the *recipe* — it isn't,
proven directly above. The gap is that this repo carries no Form-native tool to turn that resolved,
already-correct evaluation into a **flattened `.tbl` artifact** at all, for any recipe, old or new tags.
`flatten/README.md:11-15` names this directly as the standing, undone decision ("the flatten must be
fkwu-self-derivable... until that self-derivation is proven, T_flat is a flagged crutch"), and
`receipts/2026-06-29-cursor-seed-p1-form-eval-reducer.md`'s own "Where the circle currently breaks"
section lists both fkwu-native flatten routes as still blocked as of 2026-06-29, with the missing piece
this receipt confirms head-on: the serializer, not the tree-builder.

## What was NOT done, and why (per the task's own stop conditions)

- **No bash, no python, no hand-patched bytes were used or attempted** for the actual regeneration. The
  only workaround available — writing a Form-native `fks-table-file` from scratch to match the `nf fn[]
  nr nodes×4 ns strings` format — is a materially larger, riskier undertaking (a new serializer touching
  the exact byte contract every existing `.tbl` in this repo depends on) than this task's scope, and the
  task's own exit criteria class this as a "stop and report" case: *"If regenerating requires anything
  beyond Form-native tooling... — STOP, don't do it, report exactly what's missing."* What's missing is
  named precisely above.
- **`proof/four-way-run.tbl` is untouched.** `git status` is clean; the baseline verdict (0, FOUR-WAY) is
  exactly the number it was before this receipt started.
- **`runtime/fkwu-uni.c`'s tag 30/31 dispatch blocks were not touched.** Since the table could not be
  regenerated, there is nothing new to verify removing them against — the prior receipt's finding stands:
  removing them breaks `proof/four-way-run.tbl` as currently committed.
