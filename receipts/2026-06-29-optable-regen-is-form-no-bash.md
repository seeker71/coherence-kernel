# Optable regeneration is pure Form — the bash wrapper composted

**Date:** 2026-06-29
**Touches:** `flatten/gen-source-walker.sh` (DELETED),
`flatten/gen-source-walker.fk` (new, the Form splice driver),
`flatten/gen-source-walker-table.fk` (the generator — final action now writes
the header itself), `runtime/fkwu-optable.h` (GENERATED — re-synced to the ONE table)

## The smell removed

The source-walker optable (`runtime/fkwu-optable.h`) was already generated by a
**Form** recipe (`gen-source-walker-table.fk`) from the ONE op table (`flt-ops`
in `form-flatten.fk`). The only non-Form glue left was a 28-line bash wrapper,
`gen-source-walker.sh`, doing three things: (1) `grep`/`awk`/`sed` to slice the
`flt-ops` defn out of `form-flatten.fk` and splice it ahead of the generator,
(2) run the combined source on the bootstrap kernel, (3) redirect stdout into the
header. Three shells of host glue around a body that is otherwise Form.

## The fix — the regeneration is Form file-io, end to end

The source-walker runtime already carries `write_file_text` (host-io tag 104:
`open(path, O_WRONLY|O_CREAT|O_TRUNC, 0666)` + write) — confirmed live in
`fk_sparse` and in the committed `fk_optab`. So the header can land as a Form
**effect**, not a shell capture.

1. **`gen-source-walker-table.fk`** — final form changed from `(header)` (a value
   the shell redirected) to `(write_file_text "runtime/fkwu-optable.h" (header))`.
   The recipe IS the whole regeneration. (This also sidesteps a real fragility:
   `--src`'s string-root printer does **not** recurse through a top-level `(do …)`
   to a final string-returning call — the old `(do … (header))` root printed the
   *interned-string id* (`1897`) instead of the header text, so the stdout-capture
   path was already broken on the current fkwu. The write-as-effect path has no
   such dependence on root-string detection.)

2. **`gen-source-walker.fk`** (new) — the splice in **pure Form**: `read_file`s
   `form-flatten.fk`, slices the `flt-ops` defn by `str_find`/`substring` (the same
   span the old `awk` derived: from `(defn flt-ops ()` to the next top-level
   `\n(defn `), `read_file`s the generator, strips its leading `(do`, and
   `write_file_text`s a self-contained combined driver to `/tmp`. No `grep`, no
   `sed`, no `awk`, no `echo`, no redirect.

Regeneration, zero bash, from the repo root:

```
fkwu --src flatten/gen-source-walker.fk            # splice  -> /tmp/gen-source-walker-combined.fk
fkwu --src /tmp/gen-source-walker-combined.fk      # generate-> runtime/fkwu-optable.h
```

`fkwu --src` is single-file with no include, so the cross-file `flt-ops` splice is
genuinely two passes — both pure Form recipes. The combined driver is a throwaway
under `/tmp`, never committed.

## Gate

- **Form-native regen runs:** both calls succeed; `runtime/fkwu-optable.h` is
  written by the recipe itself.
- **Idempotent / fixed point:** running the two-call sequence a second and third
  time produces a byte-identical header (`diff` = 0) — the generator is a fixed
  point over the current `flt-ops`.
- **Build + run:** `cc -O2 runtime/fkwu-uni.c` builds clean; `fkwu --src` on
  `(head (list 11 22 33))` → `11`.

## One honest difference — the committed header was stale, now healed

The regenerated header is **not** byte-identical to the previously-committed
`fkwu-optable.h`. The diff is exactly two things, both the generator faithfully
reflecting the ONE table:

- `+ { "cuda_matvec", 0, 232 },` — `cuda_matvec` (tag 232, implemented in the
  runtime at `t == 232`) was added to `flt-ops` in commit `cfeabdd` (Windows RTX
  CUDA) but the header was **never regenerated** (the manual bash step was
  skipped, and as noted above the `(do)`-root capture was broken anyway). The
  committed header was missing a row that is in the source table — drift. The op
  is now registered in the source-walker optable.
- a stray trailing blank line in the old committed header (a shell-capture
  artifact the generator never emits) is gone.

Per the body's discipline — the ONE table is truth, drift is healed not preserved
— the committed header is now the faithful generator output over current `flt-ops`.

## Honest floor

Four-way / platform receipt rows: **pending**. This is a bootstrap-kernel dev tool
(manual regen, not on the CI/build path); the gate here is the byte-fixed-point +
`cc` build + `--src` run on the local c-bootstrapped fkwu. The remaining host
process is `fkwu` itself (the c-bootstrapped runtime) invoked twice — no go, rust,
clang, bash, or python in the regeneration path.
