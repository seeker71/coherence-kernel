# Receipt — standing the seed, stone 1: fkwu runs Form SOURCE directly on Windows (2026-06-29)

**The chosen next move:** stand the seed — so the `.fk` body runs as Form on Windows. The keystone is the
circularity (to run arbitrary Form you need `form-eval-cli-loop` flattened; to flatten it you need the full
flattener; the flattener needs the full grammar to flatten itself). The README names the way through: *"a minimal
flatten baked into `runtime/fkwu-uni.c`"* — a one-time bootstrap seed (like the `cc` command) that parses Form
source into the node format, used to flatten the FORM flattener so it self-hosts, then dropped. This is **stone 1**.

## Witnessed native on Windows 11 (`fkwu --src <file.fk>`)

fkwu now reads a `.fk` **source file**, parses the s-expr to the same node format `fk_run` loads, and runs it —
no hand-flattening, no numeric table, no Mac seed:

```
(add 40 2)                          -> 42
(if (le 1 2) (add 40 2) 99)         -> 42
(add (sub 50 8) (mul 0 7))          -> 42
(if (eq 3 3) (div 84 2) 0)  ; +cmt  -> 42
```

A recursive-descent parser (`fk_sparse`) handles whitespace + `;` comments, integer literals (incl. negative),
and the core ops `add/sub/mul/div/mod/le/eq/if` via a small name→tag table mirroring `flt-ops`. The parsed nodes
run through the unchanged `fk_walk`. This is the bootstrap seed's first stone: **fkwu executes Form source on
Windows** — the thing that was "impossible without the seed" now does the simplest case.

## What landed

- `runtime/fkwu-uni.c`: `fk_run_src` + the s-expr parser, dispatched by a new `--src` mode (`fkwu --src file.fk`).
  Stone 1 grammar = literals + binary/ternary core ops. Platform-neutral C (binary-safe `open` on Windows).
- No regression: the numeric-table path (`fkwu table.flat ...`), the JIT dispatch, and all sensors
  (`sense_sensors` → `IASVMS` etc.) still run.

## The honest floor — this is stone 1 of the seed, not the whole seed

- **Stands now:** Form source with literals + `add/sub/mul/div/mod/le/eq/if`, nested, with comments, run native on
  Windows from a `.fk` file.
- **Next stones** (each its own witnessed step) toward flattening `form-flatten.fk` so the Form flattener self-hosts:
  1. **bare symbols → args** (tag 2) + **`defn`** (the fn table) + **single-arg `call`** (tag 7) → run a recursive
     recipe from source (e.g. the `f(n)` accumulate, witnessed earlier only hand-flattened).
  2. **`let`** (frame slots, tags 109/110/111) + **`do`** (tag 69) + **multi-arg calls** (tag 12, frame packing).
  3. **strings + lists** (the string pool, `cons`/`head`/`tail`) — what `form-flatten.fk` itself needs.
  When stone-N parses the grammar `form-flatten.fk` uses, it flattens the Form flattener once → the **Form**
  flattener self-hosts and the C seed parser is dropped (its telos met, like clang after `form-asm`).
- **The fast alternative, unchanged:** the Mac's **platform-neutral** seed already stands; committing it unlocks
  the *full* grammar on Windows immediately (the stones are the toolchain-free path; the Mac seed is the shortcut).

## What this is, honestly

The C parser is the **bootstrap seed**, not body logic that stays in C — its purpose is to flatten the Form
flattener, then retire. Stone 1 proves the path is real on Windows: the kernel turns source into runnable nodes
itself. Each next stone widens the grammar until the Form flattener runs; then Form flattens Form, and the body
runs as Form here with no C in the loop.

## Reproduce

```
gcc -O2 -o fkwu.exe runtime/fkwu-uni.c -lws2_32 -lwinmm -lavicap32 -luser32 -lwlanapi -lbthprops
printf '(if (le 1 2) (add 40 2) 99)\n' > r.fk && ./fkwu.exe --src r.fk   # -> 42
```
